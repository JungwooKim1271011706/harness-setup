# CONTEXT — 도메인 용어집

> 프로젝트 도메인·하네스 운영 용어의 단일 정의. 용어 충돌·모호함 발견 시 여기서 확정한다.

## 하네스 운영 (GitLab 봇 연동)

### Work Item (WI)
GitLab 이슈로 등록되는 단일 작업 단위. WI 번호로 설계→생성→리뷰→머지 전 과정을 추적한다. 사내 4-Phase 워크플로의 기본 단위.

### WI as oracle (WI = 검증 기준)
리뷰모드의 검증 기준(oracle)은 **Work Item 본문**이다. 봇이 생성한 코드가 아니라 WI에 적힌 기대 행위를 정답으로 삼는다. 봇 코드를 기준으로 삼으면 잘못된 구현이 "통과"로 굳으므로 금지. WI는 설계→생성→리뷰를 잇는 단일 진실 소스(SSOT)이며, 누가 설계했든 WI만 완전하면 누구나 리뷰 가능하다.

### 설계모드 (Design Mode)
로컬 v3.2 하네스의 진입 모드 중 하나. 현행 풀사이클의 **앞부분만** 재사용한다(0단계 → office-hours → grill → co-plan → 설계패널 게이트). developer를 호출하지 않고, 종료 산출물로 **WI 본문 명세**를 출력한다. 코드 생성은 GitLab 봇이 담당(사내 Phase2). 사용자가 "설계모드" 프롬프트로 수동 트리거한다.

### 리뷰모드 (Review Mode)
로컬 v3.2 하네스의 진입 모드 중 하나. 현행 풀사이클의 **뒷부분만** 재사용한다(tester → /review → /codex review → /cso). 봇이 만든 MR 브랜치를 로컬 checkout한 상태가 입력 전제. WI 본문을 oracle로 삼아 검증하고, PASS여도 Approve 권고를 출력하지 않는다(증거 리포트만 제공, Approve 여부는 사람 책임). FAIL이면 MR 댓글용 수정지시 + BotAct 라벨 안내를 출력한다. 머지(Bundle Merge)는 봇이 담당(사내 Phase4). 사용자가 "리뷰모드 + WI번호" 프롬프트로 수동 트리거한다(자동 감지 없음).

**WI 단위 통합검증 (멀티레포)**: 검증 단위는 MR이 아니라 **WI**다. WI 하나에 모듈별 MR이 여러 개 걸리며(예: tocServer MR + tocFramework MR), 모듈 간 의존(예: tocServer→tocFramework) 때문에 따로 보면 통합 동작을 못 본다. WI에 걸린 모든 MR을 일괄 checkout(§9.5 `git submodule foreach` 재사용)해 통합 빌드·기동 후 묶어서 검증하고, 전부 PASS여도 Approve 권고를 출력하지 않는다. 증거 리포트(테스트 결과·리뷰 발견·사람 스크립트 통과표)만 제공하고 정지한다. Approve 여부는 사람 책임(A2). tester-runtime의 "통합+전체회귀 1회"(P1)와 정합.

**입력 수집 (orchestrator)**: 리뷰모드 진입 시 orchestrator가 필수 입력을 수집한다 — ① WI 번호(oracle 식별 + FAIL 카운트 키) ② MR 번호들(checkout 대상) ③ 브랜치명 또는 base(.gitmodules 자동조합 가능 시 생략) ④ WI 본문(oracle 텍스트). glab/MCP 없으면 ④ WI 본문은 사용자가 붙여줘야 하고, glab 있으면 `glab mr list`/이슈 조회로 자동 수집.

### 리뷰 FAIL 카운트 (WI별, L2)
같은 WI가 봇 핑퐁(MR 댓글→BotAct→봇 재작업→재리뷰)으로 무한 반복되지 않도록, WI별 리뷰 FAIL 횟수를 체크포인트(`~/.gstack/projects/{slug}/checkpoints/review-WI{N}.json`)에 영속한다. `/clear`·`/compact`·재접속·봇 대기 등 세션 끊김에도 사람 기억이 아니라 이 파일이 권위 소스(C6 메커니즘 재사용). N회 FAIL 시 "봇이 못 고침 = 설계(WI) 결함 의심 → 설계모드로 복귀해 WI 재작성 권고"로 진단(FAIL 3분기의 설계결함 분기와 연결).

### 모드 간 결합 = WI 매개 느슨결합
설계모드와 리뷰모드는 서로 직접 통신하지 않는다. WI(GitLab)를 매개로 느슨하게 결합된다. 현행 풀사이클(로컬 단독 0→1)도 그대로 유지되며, 세 모드가 같은 agent를 공유하고 orchestrator가 진입·종료점만 분기한다(새 agent 없음).

### STP 지향 (Straight-Through Processing)
설계모드 WI의 품질 목표는 봇 Phase1(Clarify/BotWait)을 **건너뛸 만큼 완전**한 것이다(§3.1). 봇이 되묻지 않고 바로 Phase2 생성에 들어가게 한다. 봇 Phase1은 설계모드를 안 거친 WI용 예외 경로로만 남는다. 단 봇이 되물을지는 봇 구현에 달려 통제 밖이므로 "지향"이지 "보장"은 아니다.

### 인간 책임 경계 (리뷰모드 산출물)
리뷰모드는 "Approve 권고"를 하지 **않는다**(§4 "Approve 버튼=네 코드, 책임 100% 인간"). 하네스는 증거(테스트 결과·리뷰 발견)만 제공하고, **Approve 여부는 명시적으로 사람 몫**이다. glab 자동 Approve는 영구 금지. 하네스 PASS가 인간 검증을 대체하지 않는다.

### 사람 검증 스크립트 (옆집 아저씨 기준)
리뷰모드가 출력하는 "사람이 직접 확인할 것"은 추상 체크박스가 아니라 **도메인 무지자도 따라 실행 가능한 단계별 스크립트**여야 한다(URL·계정·클릭 대상·기대 결과 명시). 추상 체크리스트는 무지성 통과를 유발하므로 금지.
- **값 출처 (가)+(다) 혼합**: URL·라우트·클릭 대상·페이지 경로는 봇 MR 코드와 로컬 기동에서 자동 추출(다). 테스트 계정·샘플 데이터는 프로젝트 테스트 시나리오 파일(가, rules처럼 프로젝트별, 민감하므로 gitignore)에서 읽는다. 파일 없으면 `<테스트ID 입력>` 플레이스홀더로 남긴다.

### 리뷰모드 = 둘 다 검증 (하네스 JUnit + 사람 스크립트)
리뷰모드는 두 가지로 검증하고 둘 다 통과해야 한다.
1. **하네스 JUnit**: tester가 WI oracle 기준으로 검증 테스트를 생성·실행(skipTests temp-override 재사용) + /review + /cso. 봇 MR에 이미 테스트가 있으면 그것도 실행.
2. **사람 스크립트**: 옆집 아저씨 기준 단계별 수동 검증([[사람 검증 스크립트 (옆집 아저씨 기준)]]).

**판정**: 둘 중 하나라도 FAIL이면 FAIL → MR 댓글 + BotAct + FAIL 카운트 +1. 둘 다 PASS여야 Approve 후보(사람 최종 책임). JUnit PASS인데 사람이 깨뜨리면(또는 반대) = FAIL로 친다(약한 테스트 또는 미검 케이스 노출). 둘은 잡는 영역이 달라(JUnit=코드 로직/분기/회귀, 사람=실제 UI/통합 동작) 서로의 공백을 메운다.

### 특성화 테스트 주의 (characterization 함정)
리뷰모드가 봇 코드를 보고 테스트를 만들 때, 봇 코드의 현재 동작을 그대로 고정하면 잘못된 로직도 통과로 굳는다. 테스트 기준은 반드시 WI 본문의 기대 행위여야 한다([[WI as oracle]]). 사람 스크립트가 독립 oracle 역할을 해 JUnit 편향을 발각하는 안전망이 된다.
