# CONTEXT — 도메인 용어집

> 프로젝트 도메인·하네스 운영 용어의 단일 정의. 용어 충돌·모호함 발견 시 여기서 확정한다.

## 하네스 운영 (GitLab 봇 연동)

### Work Item (WI)
GitLab 이슈로 등록되는 단일 작업 단위. WI 번호로 설계→생성→리뷰→머지 전 과정을 추적한다. 사내 4-Phase 워크플로의 기본 단위.

### WI as oracle (WI = 검증 기준)
리뷰모드의 검증 기준(oracle)은 **Work Item 본문**이다. 봇이 생성한 코드가 아니라 WI에 적힌 기대 행위를 정답으로 삼는다. 봇 코드를 기준으로 삼으면 잘못된 구현이 "통과"로 굳으므로 금지. WI는 설계→생성→리뷰를 잇는 단일 진실 소스(SSOT)이며, 누가 설계했든 WI만 완전하면 누구나 리뷰 가능하다.

### 설계모드 (Design Mode)
로컬 v3.2 하네스의 진입 모드 중 하나. 현행 풀사이클의 **앞부분만** 재사용한다(0단계 → office-hours → grill → co-plan → 설계패널 게이트). developer를 호출하지 않고, 종료 산출물로 **WI 본문 명세**를 출력한다. 코드 생성은 GitLab 봇이 담당(사내 Phase2). 사용자가 "설계모드" 프롬프트로 수동 트리거한다.

### 리뷰모드 (Review Mode)
로컬 v3.2 하네스의 진입 모드 중 하나. 현행 풀사이클의 **뒷부분만** 재사용한다(tester → /review → /codex review → /cso). 봇이 만든 MR 브랜치를 로컬 checkout한 상태가 입력 전제. WI 본문을 oracle로 삼아 검증하고, PASS여도 Approve 권고를 출력하지 않는다(증거 리포트만 제공, Approve 여부는 사람 책임). FAIL이면 MR 댓글용 수정지시 + BotAct 라벨 안내를 출력한다. 머지(Bundle Merge)는 봇이 담당(사내 Phase4). 사용자가 "리뷰모드 + WI번호" 프롬프트로 수동 트리거한다(자동 감지 없음).

**WI 단위 통합검증 (멀티레포)**: 검증 단위는 MR이 아니라 **WI**다. WI 하나에 모듈별 MR이 여러 개 걸리며(예: tocServer MR + tocFramework MR), 모듈 간 의존(예: tocServer→tocFramework) 때문에 따로 보면 통합 동작을 못 본다. WI에 걸린 모든 MR을 일괄 checkout(§9.5 `git submodule foreach` 재사용)해 통합 빌드·기동 후 묶어서 검증하고, 전부 PASS여도 Approve 권고를 출력하지 않는다. 증거 리포트(테스트 결과·리뷰 발견·사람 스크립트 통과표)만 제공하고 정지한다. Approve 여부는 사람 책임(A2). tester-runtime의 "통합+전체회귀 1회"(P1)와 정합.

**입력 수집 (orchestrator)**: 리뷰모드 진입 시 orchestrator가 필수 입력을 수집한다 — ① WI 번호(oracle 식별 + FAIL 카운트 키) ② MR 번호들(checkout 대상) ③ 브랜치명 또는 base(.gitmodules 자동조합 가능 시 생략) ④ WI 본문(oracle 텍스트). glab/MCP 없으면 ④ WI 본문은 사용자가 붙여줘야 하고, glab 있으면 `glab mr list`/이슈 조회로 자동 수집.

### 리뷰 FAIL 카운트 (WI별, L2)
같은 WI가 봇 핑퐁(MR 댓글→BotAct→봇 재작업→재리뷰)으로 무한 반복되지 않도록, WI별 리뷰 FAIL 횟수를 체크포인트(`~/.gstack/projects/{slug}/checkpoints/review-WI{N}.json`)에 영속한다. `/clear`·`/compact`·재접속·봇 대기 등 세션 끊김에도 사람 기억이 아니라 이 파일이 권위 소스(C6 메커니즘 재사용). N회 FAIL 시 "봇이 못 고침 = 설계(WI) 결함 의심 → 설계모드로 복귀해 WI 재작성 권고"로 진단(FAIL 3분기의 설계결함 분기와 연결).

### 모드 간 결합 = WI 매개 느슨결합
설계모드와 리뷰모드는 서로 직접 통신하지 않는다. WI(GitLab)를 매개로 느슨하게 결합된다. 현행 풀사이클(로컬 단독 0→1)도 그대로 유지되며, 세 모드가 같은 agent를 공유하고 orchestrator가 진입·종료점만 분기한다(새 agent 없음).

### STP 지향 (Straight-Through Processing)
설계모드 WI의 품질 목표는 봇 Phase1(Clarify/BotWait)을 **건너뛸 만큼 완전**한 것이다(§3.1). 봇이 되묻지 않고 바로 Phase2 생성에 들어가게 한다. 봇 Phase1은 설계모드를 안 거친 WI용 예외 경로로만 남는다. 단 봇이 되물을지는 봇 구현에 달려 통제 밖이므로 "지향"이지 "보장"은 아니다.

### 인간 책임 경계 (리뷰모드 산출물)
리뷰모드는 "Approve 권고"를 하지 **않는다**(§4 "Approve 버튼=네 코드, 책임 100% 인간"). 하네스는 증거(테스트 결과·리뷰 발견)만 제공하고, **Approve 여부는 명시적으로 사람 몫**이다. glab 자동 Approve는 영구 금지. 하네스 PASS가 인간 검증을 대체하지 않는다.

### 사람 검증 스크립트 (옆집 아저씨 기준)
리뷰모드가 출력하는 "사람이 직접 확인할 것"은 추상 체크박스가 아니라 **도메인 무지자도 따라 실행 가능한 단계별 스크립트**여야 한다(URL·계정·클릭 대상·기대 결과 명시). 추상 체크리스트는 무지성 통과를 유발하므로 금지.
- **값 출처 (가)+(다) 혼합**: URL·라우트·클릭 대상·페이지 경로는 봇 MR 코드와 로컬 기동에서 자동 추출(다). 테스트 계정·샘플 데이터는 프로젝트 테스트 시나리오 파일(가, rules처럼 프로젝트별, 민감하므로 gitignore)에서 읽는다. 파일 없으면 `<테스트ID 입력>` 플레이스홀더로 남긴다.

### 리뷰모드 = 둘 다 검증 (하네스 JUnit + 사람 스크립트)
리뷰모드는 두 가지로 검증하고 둘 다 통과해야 한다.
1. **하네스 JUnit**: tester가 WI oracle 기준으로 검증 테스트를 생성·실행(skipTests temp-override 재사용) + /review + /cso. 봇 MR에 이미 테스트가 있으면 그것도 실행.
2. **사람 스크립트**: 옆집 아저씨 기준 단계별 수동 검증([[사람 검증 스크립트 (옆집 아저씨 기준)]]).

**판정**: 둘 중 하나라도 FAIL이면 FAIL → MR 댓글 + BotAct + FAIL 카운트 +1. 둘 다 PASS여야 Approve 후보(사람 최종 책임). JUnit PASS인데 사람이 깨뜨리면(또는 반대) = FAIL로 친다(약한 테스트 또는 미검 케이스 노출). 둘은 잡는 영역이 달라(JUnit=코드 로직/분기/회귀, 사람=실제 UI/통합 동작) 서로의 공백을 메운다.

### 특성화 테스트 주의 (characterization 함정)
리뷰모드가 봇 코드를 보고 테스트를 만들 때, 봇 코드의 현재 동작을 그대로 고정하면 잘못된 로직도 통과로 굳는다. 테스트 기준은 반드시 WI 본문의 기대 행위여야 한다([[WI as oracle]]). 사람 스크립트가 독립 oracle 역할을 해 JUnit 편향을 발각하는 안전망이 된다.

## 하네스 테스트 흐름

### 변경검증 (Change Verification)
매 변경(구현)마다 가볍게 수행하는 검증. 대상 = "A 기능이 의도대로 동작하는가 + A의 직접 사이드이펙트". 담당 = tester-backend/tester-frontend. 범위 = **변경 스코프만**(직접 호출자 1홉 + planner가 feature 문서에 적은 회귀 범위). 통합·전체회귀는 하지 않는다. cross-module 통합과 변경 무관 도메인 회귀는 [[전체회귀]]가 맡는다. 사이드이펙트 산정에 자동 N홉 호출그래프·공유테이블 자동매핑은 만들지 않는다(LLM grep 추적은 부정확+거짓확신=과설계). 정적분석이 못 잡는 간접·DB 결합은 사람(planner 회귀범위)과 주기 전체회귀가 메운다.

### 전체회귀 (Full Regression)
전 도메인 안전망. tester-runtime이 담당하며 cross-module 통합 + 변경 무관 도메인까지 전체 스위트를 1회 돈다. **매 구현마다 강제하지 않는다**(과거엔 강제였고 그게 과부하의 원인이었다). 호출 경로는 둘뿐: ① 사용자 수동("회귀 돌려") ② [[전체회귀 부채]] 권장 수락. 실행해 PASS하면 부채가 리셋된다.

### 전체회귀 부채 (Full Regression Debt)
마지막 전체회귀 PASS 이후 쌓인 미검 회귀 위험의 누적량. state 파일 `~/.gstack/projects/{slug}/regression-debt.json`에 영속(기존 체크포인트 패턴 재사용, repo 밖이라 비공유). **{slug}는 `eval "$(~/.claude/skills/gstack/bin/gstack-slug 2>/dev/null)"`으로 산정(리뷰모드 체크포인트와 동일 메커니즘)**. 스키마 = { last_full_regression: {sha, ts}, commits_since: [{sha, modules, ts}] }. 부채 누적은 **제품 코드 모듈(tocServer/tocProcess/tocFramework)을 터친 커밋만** 카운트한다(문서·.claude만 바뀐 커밋은 제외). 모듈 터치 판정 = `git diff --cached --name-only` 경로 첫 세그먼트 매핑.

**2트리거 (강력권장으로 격상)**:
1. 마지막 전체회귀 후 커밋 ≥ 5 (N=5, 느슨 — 전체회귀가 무거워 자주 돌리지 않으려는 의도)
2. tocFramework 변경 = 즉시 격상, 리셋 전까지 유지 (양쪽 영향, tocFramework/backend.md "양쪽 회귀 필요"를 인코딩)

임계 미만이면 정보 1줄(📊)만, 트리거 hit면 강력권장(⚠). 누적 도메인 카운트 트리거는 모듈이 3개뿐이라 framework 룰과 중복이라 두지 않는다.

### 전체회귀 부채 안내 = 비차단 단방향 통지
부채 안내는 finalizer가 커밋 **직전** 자기 완료 리포트 안에 텍스트 1블록으로 출력하고 **그냥 진행한다**. AskUserQuestion·멈춤·답 대기 금지. "회귀 돌릴까요?"식 질문 금지. 커밋은 부채 상태와 무관하게 무조건 완료된다. 사용자는 읽고 나중에 수동 실행하든 무시하든 자유. 이 불변식이 깨지면(질문/차단 끼우면) 매 커밋 응답 마찰이 부활해 소프트 안내가 하드 게이트로 변질되므로 절대 금지. git 훅이 아니라 텍스트 출력이다(제품 repo·서브모듈 미변경).

### L1 컨텍스트 기동 / L2 풀 런타임 기동
"기동 검증"은 이 프로젝트에서 2층이다.
- **L1 컨텍스트 기동**: Spring ApplicationContext 로드 = bean/config wiring 검증. context 로드 테스트 실행으로 확인. 가볍지만 **라이브 DB(MariaDB) 필요**. tester-backend가 변경 스코프 테스트에 context 로드 테스트가 있으면 보고, 없으면 컴파일까지만(공백 인정). [[변경검증]] 책임.
- **L2 풀 런타임 기동**: Tomcat WAR 배포 + HTTP 요청 스모크. 무겁고 환경의존(Tomcat+DB+conf 실값). 매 변경 강제하지 않고 [[전체회귀]] 부채 트리거 또는 사람 검증([[사람 검증 스크립트 (옆집 아저씨 기준)]] 재사용, oracle=요구사항)으로 미룬다.

### 회귀 oracle 이원화
전체회귀의 두 경로는 검증 기준(oracle)이 다르다.
- **자동(tester-runtime)**: oracle = 기존 테스트 스위트. "변경 무관 도메인이 안 깨졌나"를 본다. 새 요구사항 불필요(기대값이 코드에 박힘).
- **사람(옆집아저씨)**: oracle = feature 문서 요구사항. "변경이 의도대로 + 실제 UI/통합 동작"을 본다. feature 문서는 신규기능·고복잡도 트랙에만 생성된다.

사람 몫의 **실기동 UI 부분은 `/qa-only`(gstack, report-only)가 부분 자동화**한다. UI 트랙에서 servable URL 확보 시 tester-frontend PASS 직후 클릭스루로 자동 점검 → PASS 흐름은 사람E2E 점검표에서 차감, 발견은 developer-frontend로(차단형, /review 발견과 동일 취급). URL 부재·앱 미기동이면 스킵하고 사람이 본다(best-effort, L2 풀 런타임은 무겁고 환경의존). 자동수정 변형 `/qa`는 거버넌스 충돌(수정=developer)이라 금지 — report-only만 쓴다.

### 반복 ≠ 신뢰 (재실행/교차검증 원칙)
검증에서 confidence는 "같은 걸 N번"으로 안 늘어난다. 정보가 느는 건 두 경우뿐: ① **상태가 바뀜**(코드 변경 후 재실행=회귀) ② **종류가 다름**(단위→통합, claude→codex, 렌즈 A→렌즈 B). 이 원칙이 하네스 여러 결정의 근거다.
- **테스트 실행은 결정적 게이트**: 같은 코드+같은 스위트 재실행은 정보 0(1회=N회). tester가 여러 번 도는 건 "같은 걸 N회"가 아니라 "바뀐 상태마다 1회"(developer 수정 루프)다. 코드 안 바뀌면 재실행 안 한다. PASS 후 tester-runtime을 또 강제하지 않는 것도 같은 이유(종류가 다른 전체회귀는 정당, 같은 재실행은 무의미).
- **검증 의견은 비상관 소스로 늘린다**: 7a∥7b(claude∥codex), /review∥/codex review = 비상관 2소스라 정보가 는다. claude가 같은 테스트를 또 돌리는 건 의견 추가가 아니다(결정적 게이트 재확인일 뿐).
- **설계패널 critical 교차검증 = perspective-diverse**: 같은 refute 프롬프트 3회(고상관)가 아니라 서로 다른 3렌즈(존재성/발현가능성/맥락 — 비상관)로 검증한다. ≥2 렌즈 반증 시 폐기. 같은 프롬프트 N회는 토큰만 N배 쓰고 confidence는 안 사는 함정.

**단순수정 oracle 구멍 (수용된 잔여 위험)**: 단순수정 트랙은 feature 문서가 없어 사람 L2 oracle이 빈약(원본 요청뿐). 단 단순수정은 1~2줄·framework 미변경이라 부채가 N=5까지 조용해 L2가 거의 안 뜨고, 변경검증으로 충분하므로 위험이 작아 수용한다.
