# Pair-Impl — 함께 만드는 구현

AI 혼자 전체를 구현하는 대신, 핵심 로직마다 멈추고 유저와 대화하며 진행한다.
목표는 유저가 "내가 만든 코드"라는 감각을 유지하는 것이다.

## 호출 시 받는 컨텍스트

오케스트레이터가 아래 정보를 전달한다:

- **plan**: 구현할 기능 계획 (planner 산출물 또는 co-plan 합의 spec)
- **domain**: `frontend` | `backend`

## Step 1 — 핵심 로직 vs 보일러플레이트 분류

plan을 읽고 구현 단위(함수/메서드/컴포넌트)를 두 그룹으로 분류한다.

### 핵심 로직 (멈추고 질문)
아래 중 하나라도 해당하면 핵심 로직이다:
- 두 가지 이상 구현 방법이 있었는데 하나를 선택한 경우
- 성능/에러 처리에서 비자명한 결정이 있는 경우
- 비즈니스 규칙이 코드에 직접 반영되는 경우
- N+1, 트랜잭션 경계, 동시성 등 주의가 필요한 패턴

### 보일러플레이트 (통과)
- DTO/타입 필드 추가
- Lombok, 어노테이션 설정
- 단순 CRUD (이름에서 기능이 다 보이는 것)
- 라우터/컨트롤러 배선
- import 정리

분류 결과를 먼저 보여준다:

```
🔍 구현 분류

핵심 로직 (같이 볼 것):
  1. [함수명] — [왜 핵심인지 한 줄]
  2. [함수명] — [왜 핵심인지 한 줄]

보일러플레이트 (통과):
  - [목록]

핵심 로직 N개 순서대로 진행할게.
```

## Step 2 — 핵심 로직 단위 구현 루프

각 핵심 로직마다 아래 루프를 반복한다:

```
[N/전체] [함수명] 구현 시작
  ↓
developer에게 해당 함수만 구현 위임
  ↓
구현 결과 + 핵심 질문 제시
  ↓
유저 답변
  ↓
설명 (맞으면 확장, 틀리면 핵심만)
  ↓
다음 핵심 로직으로 이동
```

### 질문 출력 형식

```
⚙️ [N/전체] [함수명]

[구현 코드 — 핵심 부분만 발췌, 전체 아님]

여기서 [결정 포인트]를 [이렇게] 했어. 왜 그랬을 것 같아?
```

### 질문 방향

| 결정 유형 | 질문 방향 |
|-----------|-----------|
| 알고리즘 선택 | "다른 방법도 있었는데 왜 이걸?" |
| 에러 처리 | "여기서 예외가 나면 어떻게 돼?" |
| 성능 결정 | "이렇게 안 하면 어떤 문제가 생겼을까?" |
| 비즈니스 규칙 | "이 조건이 뭘 의미하는 것 같아?" |

어떤 답변이든 유효하다. "모르겠는데"도 괜찮다.

### 설명 출력 형식

- **맞으면**: "맞아. [1~2문장 확장 설명]"
- **틀리거나 모르면**: "핵심은 이거야. [2~3문장 설명]"

설명은 5문장을 넘지 않는다.

## Step 3 — 보일러플레이트 일괄 구현

핵심 로직 루프 완료 후, 보일러플레이트를 한 번에 구현한다.
유저에게 알리기만 하고 질문 없이 진행:

```
📦 보일러플레이트 일괄 구현 중... (질문 없이 진행)
```

## Step 4 — 완료 보고

```
✅ 구현 완료

핵심 로직 [N]개 페어 구현 완료.
보일러플레이트 일괄 구현 완료.

[변경 파일 목록]
```

오케스트레이터에게: **"pair-impl 완료. tester 호출 가능."** 라고 전달한다.

## 유저 컨트롤

- **"skip"** — 현재 질문 건너뛰고 설명만 제공 후 다음으로
- **"그냥 다 짜줘"** — pair-impl 중단, 나머지를 일반 developer 방식으로 구현
- **"다시"** — 직전 설명 다시 제공
