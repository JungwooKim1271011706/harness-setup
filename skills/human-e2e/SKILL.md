# human-e2e — 사람 E2E 점검표 (필드 SSOT)

자동(tester)이 못 본 표면을 사람이 확인한다. **이 파일이 점검표 필드·출력 형식의 SSOT** — finalizer(서브에이전트, Skill 호출 불가)는 이 파일을 Read해 같은 형식으로 렌더한다. 사용자는 `/human-e2e`로 아무 때나 직접 호출.

## 불변식
- **비차단.** 점검표는 통지다 — 커밋·진행을 막지 않는다(finalizer 불변식 승계).
- 민감값(계정/PW/토큰) 평문 금지 — `<…입력>` 플레이스홀더 또는 `review/scenarios.local.md` 참조 지시문(finalizer 절차의 gstack-redact 가드와 동일 기준).
- record는 **사용자 보고 verbatim 존중** — 관찰값을 다듬거나 추측으로 보완하지 않는다(모르면 빈 칸).

## 모드 1 — render (점검표 생성)

수집(변경 표면·자동 커버·oracle)은 finalizer `## 사람 E2E 점검 안내` 절차 1~3과 동일 기준. 직접 호출 시: 변경 표면 = `git diff --name-only`(미커밋) 또는 최근 커밋, 자동 커버 = 최근 tester 보고/feature 문서, oracle = feature 문서 요구사항(없으면 원 요청).

### 필드 스키마
- **A 헤더**: 기능(feature 문서명|원 요청) / 워크트리·브랜치 / 변경 표면 / ⏱ 소요 / 🧰 필요 환경 / ✅ 자동 커버("다시 안 봐도 됨" 경계)
- **B 시나리오 행**: # / 무엇을 한다 / 기대 결과(✅) / ✗ 실패 힌트 / 판정(render 시 ☐) / 관찰값(render 시 빈 칸)
- **C 결과 기록**(record 산출): 수행일·수행자 / 항목별 판정(PASS·FAIL·SKIP)+관찰값 / 총평 / 실패 후속(FAIL 3분기 라우팅 제안)

### 폭 규칙 (터미널 잘림 방지)
- 표 컬럼 상한: 무엇을 한다 ~20자 · 기대 결과 ~20자 · 실패 힌트 ~15자. 초과분은 셀에 `†N` 각주 마커, 표 아래 `†N:` 줄로 뺀다.
- 표 전체가 화면 폭을 넘길 바에 행을 쪼갠다(한 행 = 한 확인 동작).

### 출력 템플릿
```
🔍 사람 E2E 점검 (비차단 — 읽고 직접 확인)
  기능: <feature 문서명 | 원 요청>
  워크트리: <경로>  ·  브랜치: <브랜치명>
  변경 표면: <모듈 N파일 요약>
  ⏱ 소요 ~<N>분  ·  🧰 필요: <환경>
  ✅ 자동 커버 (다시 안 봐도 됨): <tester 실측 PASS 목록>

  [준비] (1회성 — 없으면 생략)
    - <의존 기동·시드>
  [빌드/실행]   (CLAUDE.md Build & Run 인용 — 추측 금지, 복붙 가능하게)
    $ <명령>

  👁 사람 E2E 필요 (oracle: <출처>):
  | # | 무엇을 한다 | 기대 결과 (✅) | ✗ 실패 힌트 | 판정 | 관찰값 |
  |---|------------|---------------|-------------|------|--------|
  | 1 | <동작>      | <성공 신호>    | <흔한 원인> | ☐    |        |

  결과는 "/human-e2e record" 또는 "1,2 통과 / 3 실패: <본 것>"처럼 알려주면 기록한다.
```

## 모드 2 — record (수행 결과 기록)

입력 = 사용자 자유 서술("1,2,4 통과, 3은 패널 안 열림, 콘솔 TypeError"). 스킬이 C 스키마로 정규화한다.

1. **정규화**: 항목별 PASS / FAIL / SKIP + 관찰값(사용자 표현 verbatim). 언급 안 된 항목은 사용자에게 1회 확인(임의 PASS 처리 금지). SKIP은 "안 함"이지 실패가 아님 — 구별 유지.
2. **스탬프**: feature 문서 `## 수동 E2E 검증` 절에 append(문서 없으면 점검표 원문과 함께 결과만 콘솔 출력):
   ```
   ### <YYYY-MM-DD> · <수행자>
   | # | 항목 | 판정 | 관찰값 |
   ...
   총평: <전체 통과 | 부분 실패 (N PASS / N FAIL / N SKIP)>
   실패 후속: #<n> → FAIL 3분기 판정 제안: <구현|설계|환경> (<근거 1줄>)
   ```
3. **실패 라우팅**: FAIL 있으면 orchestrator FAIL 3분기 기준(구현 결함→developer / DESIGN_MISMATCH→planner+재게이트 / 환경→사용자 가이드)으로 **제안만** 한다 — 실행은 사용자/orchestrator 결정(비차단).

## finalizer 연동
- finalizer는 커밋 직전 이 파일의 **출력 템플릿·필드 스키마를 Read해 렌더**한다(Skill 도구 없음 — reference Read 패턴). 수집 절차·redact 가드·비차단 불변식은 finalizer 절이 SSOT.
- record는 커밋 후 아무 세션에서나 가능 — feature 문서가 디스크 SSOT라 세션 무관.
