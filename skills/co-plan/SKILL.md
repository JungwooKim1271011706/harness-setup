# Co-Plan — 함께 만드는 계획

AI 혼자 계획서를 써서 가져오는 대신, 유저와 단계별로 대화하며 함께 설계한다.
목표는 planner에게 전달할 합의된 spec을 만드는 것이다.

## 호출 시 받는 컨텍스트

오케스트레이터가 아래 정보를 전달한다:

- **feature**: 만들려는 기능 설명 (1~2문장)
- **domain**: `frontend` | `backend` | `mixed`
- **context**: 관련 기존 코드/API 정보 (선택)

## 진행 단계

```
Step 1: 유저 시나리오 (정상 흐름)     [필수 · 합의 루프]
Step 2: 에러/예외 시나리오             [필수 · 합의 루프]
Step 3: API 계약                       [domain=mixed 또는 frontend+backend 연동 시만 · 자동 진행]
Step 4: 클래스 설계                    [필수 · 자동 진행]
Step 5: 메서드 설계                    [필수 · 자동 진행]
```

## 각 Step 진행 방식

**Step 1~2 — 합의 루프** (사용자가 SSOT인 도메인 지식: 어떤 흐름이어야 하나, 실제 어떤 에러가 나나):

```
1. AI가 초안 제시
2. 유저 응답
3. AI 반영/수정
4. 유저 "ㅇㅇ" 또는 "넘어가자" → 다음 Step
```

**Step 3~5 — 자동 진행** (기술 실현 — 뒤에 planner → 설계패널∥codex → 사용자 승인 게이트가 검증하므로 여기서 왕복하지 않는다):

```
1. AI가 초안을 출력하고 **응답 대기 없이** 다음 Step으로 진행 (관망 가능하게 표시는 유지)
2. 유저가 어느 시점이든 개입("잠깐"·수정 지시)하면 그 Step부터 합의 루프로 전환
```

### 출력 형식

```
📐 Step N/5 — [단계명]

[Step 1이면: ASCII 흐름도 → 그 아래 단계별 번호 목록]
[그 외 Step: 초안 내용]

[Step 1~2만] 어떻게 생각해? 수정하고 싶은 부분 있어?
[Step 3~5] (질문 없이 초안 출력 → 바로 다음 Step)
```

Step 1 ASCII 흐름도 예시:

```
  ┌─────────┐     ┌──────────┐     ┌─────────┐
  │ 유저     │────▶│ 로그인    │────▶│ 대시보드 │
  │ 진입     │     │ 검증      │     │ 조회     │
  └─────────┘     └────┬─────┘     └─────────┘
                       │ 실패
                       ▼
                  ┌─────────┐
                  │ 에러안내 │
                  └─────────┘

1. 유저 진입 → 2. 로그인 검증 → 3a. 성공: 대시보드 / 3b. 실패: 에러안내
```

### 초안 작성 기준

- **유저 시나리오**: 현재 흐름 → 변경 후 흐름. **ASCII 흐름도(박스 `┌─┐│└─┘` + 화살표 `──▶ ▼`)를 콘솔에 도식화**한 뒤 단계별 번호 목록을 병기한다 — 유저가 시각적으로 바로 이해하게. 분기(성공/실패·조건)는 화살표 분기로 표현. 터미널서 즉시 읽히게 **ASCII만** 사용(mermaid·excalidraw·파일 생성 금지). 흐름이 단순 선형이면 도식 생략 가능, 유저가 "목록만" 하면 텍스트만.
- **에러 시나리오**: "~가 없으면?", "~가 실패하면?" 형태로 3~5개.
- **API 계약**: 엔드포인트, 요청/응답 shape, 에러 코드. **외부 API 응답을 매핑하는 DTO는 실제 JSON 필드명(snake_case 등)·`@JsonProperty` 매핑을 계약에 명시**하고 "JSON round-trip 테스트 대상"으로 표시한다(목킹만으로 통과하는 필드매핑 통합버그 차단 — tester-design RED 규칙과 연동).
- **클래스 설계**: 새로 만들거나 변경할 클래스/컴포넌트 목록 + 역할 한 줄.
- **메서드 설계**: 클래스별 핵심 메서드 시그니처 + 역할 한 줄.

## Step 건너뛰기

유저가 "skip" 또는 "넘어가" 하면 해당 Step을 건너뛴다(초안 출력 자체를 생략).
단, Step 1(유저 시나리오)와 Step 4(클래스 설계)는 건너뛸 수 없다.
⚠ Step 3~5의 **자동 진행은 스킵이 아니다** — 초안은 출력하되 합의 왕복만 없는 것. 건너뛰기 규칙과 별개.

## 완료 후 처리

모든 Step 합의 완료 후 아래 형식으로 spec을 정리한다:

```
✅ 합의된 spec

## 유저 시나리오
[합의 내용]

## 에러 시나리오
[합의 내용]

## API 계약
[합의 내용 또는 "해당 없음"]

## 클래스 설계
[합의 내용]

## 메서드 설계
[합의 내용]
```

이 spec을 planner에게 전달하며 "이 spec 기준으로 기술 구현 계획을 작성해줘"라고 지시한다.
오케스트레이터에게: **"co-plan 완료. planner 호출 가능."** 라고 전달한다.
