# ADR-0001: GitLab 봇 워크플로 ↔ 로컬 하네스 2모드 브리지

## 상태

Accepted (2026-06-05)

---

## 맥락

사내가 SVN→GitLab 전환하며 Claude 봇 4-Phase 워크플로를 도입했다.

| Phase | 담당 | 내용 |
|-------|------|------|
| Phase 1 | 봇 (BotWait 핑퐁) | 기획/Clarify — WI 본문 작성 |
| Phase 2 | 봇 (Max계정, 백그라운드 스케줄러) | 코드 생성, MR 생성 |
| Phase 3 | 인간 | 리뷰·검증 (도구 미제공) |
| Phase 4 | 봇 | Bundle Merge |

로컬 PC의 v3.2 하네스는 설계 중심 풀사이클 단독 개발용으로 구성되어 있다.

```
0단계 → office-hours → grill → co-plan → 설계패널 → TDD → tester → review → cso → finalizer
```

두 시스템의 결합 방식 결정이 필요한 상황이다. 봇은 코드 생성에 강점이 있지만 아래 두 약점이 있다.

- **Phase 1 빈약**: BotWait 핑퐁으로 WI 완전성 보장 어려움
- **Phase 3 떠넘김**: "철저한 로컬 검증"을 인간에게 위임하면서 도구를 제공하지 않음

---

## 결정

로컬 하네스를 봇 워크플로의 **북엔드(bookend)**로 재배치한다.

현행 풀사이클을 유지하면서, 동일 agent들을 공유하는 **2개 모드**를 추가한다. orchestrator가 진입점·종료점만 분기한다. 새 agent를 추가하지 않는다.

### 모드 정의

| 모드 | 진입점 | 종료점 | 산출물 | 코드 생성 |
|------|--------|--------|--------|----------|
| **설계모드** | 0단계 | 설계패널 | WI 본문 명세 | 봇 (Phase 2) |
| **리뷰모드** | tester | cso | 검증 보고서 | 없음 (MR checkout) |

### 핵심 결정 목록

**WI = oracle (SSOT)**
리뷰 검증 기준은 봇이 생성한 코드가 아니라 Work Item 본문이다. 설계모드와 리뷰모드는 WI를 매개로 느슨하게 결합한다. 누가 설계했든 WI만 완전하면 누구나 리뷰할 수 있다.

**D1 — 수동 트리거**
두 모드 모두 사용자 프롬프트로 시작한다. 자동 감지 없음. 로컬 하네스는 데몬이 아니고, 봇이 이미 GitLab 이벤트를 폴링하고 있다.

**G2 — 설계모드 이중 확인**
설계모드는 WI 미리보기 후 2차 확인을 받고 출력한다. 봇 트리거 비용(Phase 2 코드 생성)이 크기 때문이다.

**L2 — FAIL 카운트 영속**
WI별 리뷰 FAIL 횟수를 체크포인트에 영속한다. 봇 핑퐁 무한 루프를 방지하기 위함이다. N회 FAIL 시 설계 결함으로 진단한다. 세션 끊김(`/clear`, `/compact`, 재접속) 대비 C6 메커니즘을 재사용한다.

**M2 — WI 단위 통합 검증**
리뷰모드는 WI 단위로 멀티레포 MR을 일괄 checkout해 검증한다. 모듈 의존 방향 때문에 분리 검증이 불가능하다.

**P2 STP 지향**
설계모드 WI는 봇 Phase 1을 건너뛸 수 있는 완전성을 목표로 한다(§3.1 Straight-Through Processing).

**A2 — 인간 책임 경계**
리뷰모드는 "Approve 권고"를 하지 않는다. 검증 증거만 제공하고, 승인은 사람의 책임이다. `glab` 자동 Approve는 영구 금지한다.

**둘 다 검증 원칙**
하네스 JUnit(WI oracle 기준 생성·실행, `skipTests` temp-override) + 사람 스크립트(옆집 아저씨 상세도) — 하나라도 FAIL이면 전체 FAIL. 값 출처는 아래와 같이 구분한다.

| 값 유형 | 출처 |
|---------|------|
| URL / 라우트 | 코드·기동 자동 추출 |
| 계정 / 테스트 데이터 | 프로젝트 시나리오 파일 (gitignore 대상) |

**D4 — 수동 복붙 전제**
`glab` / GitLab MCP는 향후 옵션이다. 봇 토큰·사내망·Maintainer 권한이 미확정이므로 당분간은 수동 복붙을 전제한다.

## 결과

### 장점

- 봇의 약점(빈약한 Phase 1, 떠넘긴 Phase 3)을 하네스 강점(설계 중심, tester/review/cso)으로 보완
- 현행 풀사이클 그대로 유지
- 새 agent 추가 없음 (0)

### 단점 / 비용

- 모드 간 자동 연속 불가. 사람 개입 3곳이 남는다.
  1. WI 등록 (설계모드 → GitLab 등록 → 봇 Phase 2)
  2. MR checkout (봇 Phase 2 완료 → 리뷰모드 진입)
  3. Approve (리뷰모드 완료 → GitLab Approve → 봇 Phase 4)
- `glab` 미설치·사내망·토큰 미확정으로 당장은 수동 복붙
- 봇이 STP를 보장하지 않으면 GitLab에서 재응답이 필요할 수 있음

### 후속 작업

| 항목 | 내용 |
|------|------|
| glab/MCP 연동 | 개입 1·2 자동화 가능 (개입 3 Approve는 영구 사람) |
| 안티패턴 섹션 | 별도 작업 |
| 테스트 시나리오 파일 포맷 | 별도 작업 |
